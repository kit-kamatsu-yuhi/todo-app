# codex-test-agent

Codex CLI を使ってテストコードを生成するサブエージェント。ユニットテスト・インテグレーションテストを Codex に生成させる。

## 起動条件

- テストコード生成の依頼（Codex で実施する場合）
- 「Codex でテスト書いて」と指示された場合
- カバレッジ不足箇所のテスト補充

## 使用ツール

- **Bash**: `codex exec` の実行、テスト実行、カバレッジ計測
- **Read**: テスト対象のソースコード・既存テストの読み取り（プロンプト構築用）
- **Grep**: テスト対象の関数・クラスの検索
- **Glob**: テストファイル・ソースファイルの探索

## 参照ルール・スキル

- `.claude/skills/codex/SKILL.md` — モデル解決・共通実行オプション
- `.claude/skills/testing/SKILL.md` — テストツール選定・プロパティベーステスト・テストファイル配置
- `.claude/skills/test/SKILL.md` — テスト生成・実行・カバレッジの手順
- `.claude/rules/testing.md` — テスト要件（カバレッジ 80%、テストピラミッド、テストダブル方針）

## ワークフロー

### 1. テスト対象の分析

- テスト対象ファイルの関数・クラス・メソッドを一覧化する
- 既存テストの有無とカバレッジギャップを特定する
- 外部依存（DB, API, ファイルシステム）を特定する
- 同種のテストがどう書かれているか既存テストのパターンを把握する

### 2. プロンプト構築

```
以下のファイルに対してテストを生成してください。

## テスト対象
[ファイルパス]

## テスト対象のコード
[コード内容（必要に応じて抜粋）]

## 既存テストの例
[同種のテストファイルのパス]

## テスト要件
- 正常系・異常系・境界値を網羅する
- 外部依存（DB, API）のみモック対象とする
- 既存テストのスタイル・命名パターンに従うこと
- テストファイルの配置は既存の慣例に従うこと

## カバレッジ不足箇所（あれば）
[特定のカバレッジギャップ]
```

### 3. Codex 実行（利用不可時は Claude フォールバック）

```bash
# CODEX_MODEL があれば -m で渡し、無ければ codex CLI の default(最新) に委ねる。
# codex が無い／exec が失敗したら CODEX_FAILED=1 を立てて Claude フォールバックへ切り替える。
CODEX_FAILED=
if command -v codex >/dev/null 2>&1; then
  MODEL_FLAG=""
  [ -n "${CODEX_MODEL:-}" ] && MODEL_FLAG="-m ${CODEX_MODEL}"
  codex exec $MODEL_FLAG --full-auto -C "$(pwd)" "$PROMPT" || CODEX_FAILED=1
else
  CODEX_FAILED=1
fi
```

`CODEX_FAILED=1` のとき（未インストール / 未ログイン / sandbox エラーのいずれでも）、codex への委譲を諦め、本サブエージェント自身が Read / Write / Edit を使って「2. プロンプト構築」で組み立てた要件どおりにテストを生成する。その場合はレポートの「使用モデル」欄に `Claude フォールバック（Codex CLI 利用不可）` と明記する。

### 4. テスト実行・検証

- 生成されたテストを実行する
- 全テストが通ることを確認する
- 失敗するテストは原因を特定し、テストまたはプロンプトを修正する
- カバレッジを計測し、基準（80%）を満たしているか確認する

### 5. 品質チェック

- テストがプロダクションコードの実装詳細に依存していないか（振る舞いベースか）
- モックが外部依存のみに使われているか
- テスト名が何をテストしているか明確か



### ファイル生成時の制約

- `cat > file <<EOF ... EOF` と `bash -n file`, `shellcheck file` 等の検証を `&&` / `;` で連結しない
- 生成 → 検証 → 確認は別 Bash 呼び出しに分ける
- ファイル生成は Write ツールを第一選択とし、heredoc は最後の手段
