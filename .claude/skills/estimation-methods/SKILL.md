# 開発見積もり手法

見積もりは「当てる行為」ではなく「不確実性を管理するツール」。外れることを前提に、バッファを確保し、実績を記録継続し、精度を向上させる。

## 見積もりが難しい理由

### 未知のタスクを数値化する行為

見積もりの本質は「まだやっていないことを数値に変える行為」。仕様が不明確なまま工数を算出することもあれば、未経験の技術を前提に見積もることもある。不確実性が高まるほど見積もり値は外れやすくなる。

### 楽観バイアス

人間は最良シナリオに無意識に基準を置く。実際の開発では以下が必ず発生する:

- 仕様認識の相違が判明
- 環境構築に予想以上の時間を要する
- レビュー指摘による修正業務

### 圧縮圧力

見積もり提出後、「短縮できないか」という圧縮要求が発生する。根拠なく数字を削るとバッファなしの脆弱な計画が生まれる。

**見積もりは「予測」であって「約束」ではない。** この共通理解の構築がプロジェクト管理者の重要な責務。

## 代表的な見積もり手法

### ストーリーポイント

スクラム開発で広く採用。工数を「時間」ではなく「複雑さと不確実性の相対的大きさ」で表現する。

**単位**: フィボナッチ数列（1, 2, 3, 5, 8, 13, 21）

**具体例**:

```
基準タスク:     ユーザー一覧取得API = 3ポイント
比較軽度タスク: 検索API           = 5ポイント
複雑度高:       決済処理           = 13ポイント
```

**相対値の利点**: 個人差により「5時間」の解釈は変動するが、「あのタスクの2倍複雑」という相対評価はチーム内で統一しやすい。

**ベロシティ管理**: 毎スプリントの消化ポイント数を計測継続することで、スケジュール予測精度が向上する。

**プランニングポーカー**: 全員が同時にカードを提示し、声の大きさに左右されない独立した見積もり意見を集約する。大きなばらつきは認識齟齬の指標となり、議論で共通理解を形成する。

### 三点見積もり（PERT見積もり）

単一値ではなく、3つのシナリオから期待値を計算する手法。

| シナリオ | 意味 | 例 |
|---------|------|-----|
| 楽観値（O） | トラブル無しの最善ケース | 3日 |
| 最頻値（M） | 最も起こりやすい通常ケース | 5日 |
| 悲観値（P） | 最悪トラブルが重複するケース | 12日 |

**計算式**:

```
期待値 = (O + 4M + P) / 6
       = (3 + 4×5 + 12) / 6
       = 5.8日
```

最頻値5日より5.8日という上方修正は、悲観シナリオのリスクを織り込んだ結果。

**活用場面**: ストーリーポイントが相対的な複雑さを測るのに対し、三点見積もりは実日数・実時間の算出に適しており、ステークホルダーへの説明やスケジュール表作成に有効。

### 類推見積もり

過去の似たタスク実績をもとに見積もる手法。

```
過去実績: ユーザー管理CRUD実装 → 5日
現在タスク: 商品管理CRUD実装 → 5～6日（若干複雑のため+1日調整）
```

**精度向上の条件**: 見積もりと実績の記録継続が前提。記録がなければ「感覚頼み」に陥り外れやすくなる。

### 手法比較

| 手法 | 向く場面 |
|------|---------|
| ストーリーポイント | スクラム型開発、チームベロシティ管理 |
| 三点見積もり | 納期重視、ステークホルダー説明必須場面 |
| 類推見積もり | 過去実績蓄積、素早い概算算出時 |

## バッファの考え方

### バッファは「サボり」ではなく「リスク対策」

バッファは「予備の怠け時間」ではなく「トラブル発生時の吸収機構」。

開発で必ず発生する事象:

- 仕様認識相違の発覚と修正
- 外部API仕様変更への対応
- レビュー指摘による再作業

リスクをゼロと仮定して計画すると、1つのトラブルでスケジュール破綻する。

### MoSCoW法でスコープの優先度を整理する

圧縮要求の前に、何が「必須」で何が「あれば良い」かを整理することで交渉が容易になる。

| 分類 | 意味 | 例 |
|------|------|-----|
| **Must have** | リリース必須。欠くと配布不可 | ログイン機能、決済処理 |
| **Should have** | 重要だがリリース可能 | 通知メール、検索フィルター |
| **Could have** | あると嬉しいが無くても問題なし | ダークモード、CSVエクスポート |
| **Won't have** | 今回は実施しない（将来候補） | 多言語対応、高度な分析 |

プロジェクト開始時にステークホルダーとMoSCoW分類を合意すれば、圧縮要求時に「Should haveをフェーズ2に延期」という具体的交渉が可能。

### 圧縮要求への対応

見積もりを削るのではなく、「削るなら何を犠牲にするか」を明示する。

**スコープ削減による交渉**:

```
「工数削減は困難ですが、本機能をフェーズ2へ延期すれば3日短縮可能」
```

**リスク明示による短縮案提示**:

```
「5日を3日に短縮するにはレビューを簡略化する必要があります。
品質リスク上昇を許容いただけますか？」
```

## 見積もり精度を上げるコツ

### タスクを小さく分解する

大規模タスクほど見積もり外れは拡大する。目安は「1タスク1～2日以内」の粒度。

**分解例**（「ユーザー管理機能実装」→）:

- 一覧取得API実装
- 登録API実装
- バリデーション追加

### 見積もりと実績を記録して振り返る

見積もり提出で終わらず、実際の所要時間を記録する。スプリント振り返りで「見積もり5ポイント = 実績8ポイント相当」という差分分析を行い、次回精度を向上させる。

### 不確実性が高いタスクを早めに特定する

プロジェクト開始時に「よくわからないタスク」を明示する。不確実性の高い項目はスパイク（調査タスク）を早期に投入して不確実性を低下させ、その後に本実装見積もりをやり直す。
