# DB マイグレーション管理

$ARGUMENTS に基づいてマイグレーションを生成・検証する。

## 手順

1. **マイグレーションファイル生成**
   - 変更内容に基づいてマイグレーションファイルを生成する
   - UP（適用）と DOWN（ロールバック）の両方を含める
   - 命名規則: `YYYYMMDDHHMMSS_[description]`

2. **前方互換性チェック**
   - マイグレーション適用後も旧バージョンのコードが動作するか確認する
   - カラム追加は NOT NULL + DEFAULT を使用する
   - カラム削除は段階的に行う（1. コードから参照削除 → 2. カラム削除）

3. **大量データへの影響分析**
   - テーブルの行数を確認し、大量データ（100万行以上）の場合は特別な考慮をする
   - ALTER TABLE のロック時間を見積もる
   - 大量データのテーブルでは pt-online-schema-change / gh-ost 等のオンラインスキーマ変更ツールの使用を検討する
   - バッチ処理でデータ移行する場合のチャンクサイズを決定する
   - ピークタイムを避けた実行計画を立てる

4. **データ整合性検証**
   - 既存データに対するマイグレーションの影響を分析する
   - NULL 制約、外部キー制約の影響を確認する
   - データ変換が必要な場合のバリデーションを設計する

5. **ロールバック手順の作成**
   - DOWN マイグレーションが正しく動作するか確認する
   - データ損失のリスクがある場合は警告する
   - ロールバック時のデータ整合性も検証する

6. **インデックス設計**
   - クエリパターンに基づいて必要なインデックスを追加する
   - 大量データのテーブルでは CONCURRENTLY オプションを使用する
   - 不要なインデックスを特定する

## 出力

- マイグレーションファイル
- ロールバックスクリプト
- 適用手順書（大量データの場合はオンラインスキーマ変更の手順を含む）
