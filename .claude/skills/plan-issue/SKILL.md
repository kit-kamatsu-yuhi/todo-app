# Issue 実装計画の策定

$ARGUMENTS の Issue を分析し、以下の手順で実装計画を策定する。

> **`/plan-product` との使い分け**:
>
> | | `/plan-issue` | `/plan-product` |
> |---|---|---|
> | **起点** | 既存の開発 Issue | アイデア・プロダクト企画 |
> | **スコープ** | 1 つの Issue の実装計画 | プロダクト全体の計画 |
> | **主な作業** | UML → API → DB → テスト → タスク分解 | ニーズ検証 → 要件定義 → Issue 起票 → リリース戦略 |
> | **出力** | `plan.md`（設計・タスク分解） | プロダクト計画 + GitHub Issue 群 |

## 参照スキル

- `uml` — Mermaid 記法での UML 設計
- `api-design` — API 設計方針
- `db-design` — DB 設計方針
- `security` — セキュリティ方針
- `testing` — テスト戦略
- `ui-ux-pro-max` — UI/UX デザイン

## 手順

### 1. Issue の要件分析

- Issue 番号が指定された場合は `gh issue view` で内容を取得する
- 機能要件・非機能要件を整理する
- 受入基準を具体的なテスト条件として定義し、以下のカテゴリに分類する:
  - **E2E テスト**: ユーザーフロー全体を検証する自動テスト
  - **自動テスト**: Unit / Integration テストで検証可能な項目
  - **手動テスト**: 画面操作・視覚確認・外部連携など自動化困難な項目
- 不明点・曖昧な箇所があれば AskUserQuestion で確認する

### 2. UML 設計（Mermaid 形式）

- クラス図: 主要なエンティティとその関係
- シーケンス図: 主要なユースケースの処理フロー
- 必要に応じてユースケース図、アクティビティ図を追加

### 3. API 設計

- エンドポイント一覧（メソッド、パス、説明）
- リクエスト/レスポンスの型定義
- エラーレスポンスの定義

### 4. DB 設計

- テーブル定義（カラム、型、制約）
- リレーション（ER 図を Mermaid 形式で）
- インデックス設計

### 5. フロントエンド設計

Issue に UI を含む場合に実施する。プロジェクトで採用している FW・言語に合わせる。

- **画面・コンポーネント構成**: 画面一覧と各画面のコンポーネントツリーを整理する
- **状態管理**: どの状態をどこで持つか（ローカル / グローバル / サーバー状態）を決定する
- **ルーティング**: URL 設計とページ遷移を定義する
- **API 連携**: フロントエンドから呼び出す API エンドポイントと、データ取得・更新のタイミング（初回描画時 / ユーザー操作時 / リアルタイム等）を整理する
- **UI/UX**: `ui-ux-pro-max` skill を参照し、レイアウト・インタラクション・レスポンシブ対応を設計する
- **バリデーション**: フロント側のバリデーションルールと、サーバー側バリデーションとの責務分担を明示する

### 6. セキュリティ基準

`security` スキルを参照し、Issue に関連するセキュリティ要件を定義する:

- 入力バリデーション要件（対象フィールド・バリデーションルール）
- 認証・認可要件（必要な権限レベル・アクセス制御）
- データ保護要件（暗号化・マスキング・PII の取り扱い）
- 外部通信要件（TLS・API キー管理）

### 7. ロギング要件

- ログ出力が必要なイベント（操作ログ・エラーログ・監査ログ）
- ログレベルの指定（INFO / WARN / ERROR）
- ログに含めるコンテキスト情報（リクエストID・ユーザーID等）
- 機密情報のマスキング要件

### 8. テスト戦略（`testing` skill 参照）

- テストピラミッドに基づき、Unit / Integration / E2E の対象範囲を決定する
- 各機能の受入基準をテストケースに落とし込む（E2E / 自動テスト / 手動テストの分類を明示する）
- 入力空間が広い関数にはプロパティベーステストを検討する
- カバレッジ目標（80%以上）の達成計画を立てる

### 9. タスク分解と依存関係の特定

- 1 タスク 2 時間以内に分解する
- タスク間の依存関係を明示する
- 優先順位を付ける
- 各機能タスクに対応するテストタスクを含める

### 10. リスク分析と対策案

- 技術的リスク、スケジュールリスク、外部依存リスクを洗い出す
- 各リスクの影響度と発生確率を評価する
- 対策案を提示する

## 出力フォーマット

以下の2ファイルを `raw/issues/YYYY-MM-DD_[issue-number]/` に保存する。

### plan.md

実装計画の本体。上記手順の各セクション（要件分析・UML・API・DB・フロントエンド・セキュリティ基準・ロギング要件・テスト戦略・タスク分解・リスク分析）をまとめる。

### todos.md

タスク分解の結果をチェックリスト形式で出力する。`/start-issue` の Phase B で進捗管理に使用する。

```markdown
# タスクリスト

## 実装タスク
- [ ] タスク1: 説明（見積もり: Xh）
- [ ] タスク2: 説明（見積もり: Xh）

## テストタスク
- [ ] テスト1: 説明
- [ ] テスト2: 説明

## ドキュメントタスク
- [ ] `raw/` コンテキスト記録
```

### plan.md に含める「実行フロー」セクション

plan.md の末尾に、計画策定後の実行フローを**必ず**明記する:

```markdown
## 実行フロー

1. ✅ `/plan-issue` — 計画策定（完了）
2. ⬜ ユーザー承認 — plan.md + todos.md の内容を確認してもらう
3. ⬜ `/codex-team all` — 実装/テスト/レビュー（codex sub-agent チームで実行）
   - codex-implement + codex-test: 実装・テスト（Agent ツールで並列起動）
   - codex-review + review-agent: レビュー（Agent ツールで並列起動）
   - acceptance-criteria-agent: 受入基準 RED/GREEN 判定
4. ⬜ `/create-pr` — PR 作成（/walkthrough → changes.md → PR）
```

この実行フローセクションは plan.md のコンテキストとして残り、後続フェーズで `/codex-team` の呼び出しを確実にする。

### changes.md

実装完了後に `/walkthrough` コマンドで生成する。`/start-issue` の Phase C で `/create-pr` 経由で自動呼び出しされるため、plan.md 作成時点では生成不要。

## 出力時のコンテキスト圧迫防止

plan.md の各セクション生成後、会話には**サマリー（1-2行）のみ**を出力し、詳細はファイルに書く。
会話に plan.md の全文を貼り付けないこと。コンテキストが膨れると後続のルーティング指示が埋没する。

```
✅ plan.md を生成しました: raw/issues/YYYY-MM-DD_XX/plan.md
✅ todos.md を生成しました: raw/issues/YYYY-MM-DD_XX/todos.md

→ 次のステップ: [呼び出し元に応じた指示]
```

## 次ステップのルーティング（必須・スキップ不可）

plan.md と todos.md の出力が完了したら:

1. **ユーザーに plan.md + todos.md の承認を求める**（承認なしに次に進んではならない）
2. **ユーザーが承認したら、Skill ツールで `/codex-team all` を必ず呼び出す**

### ルーティング先

- **`/start-issue` から呼ばれた場合**: plan.md/todos.md を出力 → ユーザー承認 → 制御を返す。呼び出し元の `/start-issue` が `/codex-team all` を呼ぶ
- **単独で呼ばれた場合**: plan.md/todos.md を出力 → ユーザー承認 → **Skill ツールで `/codex-team all` を必ず呼び出す**（スキップ不可）

## 禁止事項

- plan.md/todos.md 出力後にユーザー承認なしで実装に進むこと
- `/codex-team` を経由せずに Claude が直接実装・テスト・レビューを行うこと（Agent ツール起動失敗時のフォールバックを除く）
- plan.md の「実行フロー」セクションを省略すること
