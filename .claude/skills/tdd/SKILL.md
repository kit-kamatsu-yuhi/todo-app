# TDD（テスト駆動開発）

RED → GREEN → REFACTOR サイクルの知識ベース。ガチモードでは標準適用、ノリモードでも入出力が明確な箇所にハーネスとして適用する。

## 適用基準

### TDD を適用するケース

- **純粋なロジック・計算処理**: 税計算、ポイント還元、料金プラン判定等
- **データ変換・バリデーション**: JSON→内部モデル変換、フォーム入力検証、型変換
- **既存バグ修正**: バグを再現するテストを書いてから修正する（回帰防止）
- **リファクタリング**: 既存テストをガードレールにして安全に構造変更する

### TDD を適用しないケース

- **UI/UX・見た目の調整**: レイアウト、色、アニメーション等のVibe領域
- **新規プロトタイピング**: 仕様が未確定で入出力の定義が困難
- **インフラ・ネットワーク依存処理**: モック構築のコストが実装を上回る場合
- **AI生成コンテンツの評価**: 正解の自動判定が困難な領域

## RED → GREEN → REFACTOR サイクル

### RED（テスト失敗）

1. 受入基準から1つの振る舞いを選ぶ
2. その振る舞いを検証するテストを書く
3. テストを実行し、**失敗することを確認する**（RED状態）
4. テストが意図通りの理由で失敗しているか確認する

### GREEN（テスト成功）

1. テストをパスする**最小限の実装**を書く
2. 美しさやパフォーマンスは後回しにする
3. テストを実行し、**成功することを確認する**（GREEN状態）
4. 既存テストが壊れていないことも確認する

### REFACTOR（リファクタリング）

1. テストが全てGREENの状態でリファクタリングする
2. 重複の除去、命名の改善、構造の整理を行う
3. リファクタリング後もテストが全てGREENであることを確認する
4. テスト自体のリファクタリングも行う

## ガチモードでの TDD ループ

`/implement-gachi` の Phase B で以下のループを回す。

### 受入基準の種別と RED/GREEN 判定方法

| 種別 | テストコード作成 | RED/GREEN 判定方法 |
|------|:---:|---|
| 自動テスト（Unit/Integration） | ○ | テストコードの実行結果で判定 |
| E2E テスト | ○ | テストコードの実行結果で判定 |
| セキュリティ基準 | × | ソースコードと基準を照合して判定 |
| 手動テスト | × | ユーザーに確認を求めて判定 |
| その他 Issue 固有の基準 | × | ソースコードと基準を照合、またはユーザーに確認して判定 |

### ループ手順

1. **RED 判定**: acceptance-criteria-agent で受入基準の充足状況を確認し、未達成の基準を特定する
2. **テスト作成**: 未達成の受入基準のうち、自動テスト・E2E テストに該当するものをテストコードに変換する（テスト失敗 = RED 状態）
   - テストは codex-test エージェントと Claude で併用して作成する
3. **GREEN**: 実装してテストをパスさせる。acceptance-criteria-agent で受入基準の達成を確認する
   - 実装は codex-implement エージェントを活用する
4. **レビュー**: review-agent で多角的レビュー（セキュリティ基準含む）を実施し、codex-review でコードレビューを実施する
   - セキュリティ基準は plan.md の基準を参照
   - security スキルの知識も参照する
5. **REFACTOR**: レビュー結果を踏まえ、テスト全グリーンを維持しながらリファクタリングする
6. 受入基準が全て GREEN（acceptance-criteria-agent 確認済み）になるまでループする

Sub agent が使える場合はそれぞれ並列で動かす。
Claude Code 環境の場合は Agent Teams で行う。

acceptance-criteria-agent は「レビュー」ではなく「RED/GREEN 判定」として機能する。受入基準の RED（未達成）/ GREEN（達成）が Phase B の進行を制御する。

### ループの終了条件

- acceptance-criteria-agent で全受入基準が GREEN になっている
- review-agent の指摘がすべて解消されている
- ユーザーが実装完了を承認する

### ループの制限

- 最大5回を目安とし、収束しない場合はユーザーに相談する

## ノリモードでの TDD（ハーネス）

`/implement-nori` の Phase 3 では、上記「適用するケース」に該当する箇所のみ TDD を適用する:

- バリデーション関数、データ変換ロジック、ビジネスルール等に RED→GREEN→REFACTOR を適用する
- UI/見た目/プロトタイプ的な箇所は TDD なしで進行する
- 動作する状態になった後、TDD未適用箇所のテストを逆スペックとして生成する

## テスト設計の原則

- テスト名に「何を」「どんな条件で」「どうなるか」を含める
- 1テスト1アサーションを原則とする（関連アサーションのグループ化は許容）
- テスト間の依存を排除する（各テストが独立して実行可能）
- テストファイルはソースコードと同じディレクトリに配置する（Agentic Search最適化）
- 外部依存（DB, API, ファイルシステム）のみモック対象とする
