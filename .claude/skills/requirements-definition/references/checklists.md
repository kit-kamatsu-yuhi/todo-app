## 要件定義チェックリスト

### プロジェクト開始時

- [ ] プロジェクトの目的・ゴールが明確か
- [ ] 主要なステークホルダーを特定したか
- [ ] プロジェクトのスコープ（範囲）を定義したか
- [ ] 成功基準を設定したか
- [ ] 制約条件を把握したか

### 要求収集フェーズ

- [ ] 全てのステークホルダーにヒアリングを実施したか
- [ ] 現行業務の分析を行ったか
- [ ] 潜在的なニーズを掘り下げたか
- [ ] 要求の背景・理由を確認したか
- [ ] 矛盾する要求を特定・解決したか

### 要件定義フェーズ

- [ ] 機能要件を漏れなく定義したか
- [ ] 非機能要件を定義したか
- [ ] 優先順位をつけたか
- [ ] 実現可能性を検証したか
- [ ] 曖昧な表現を排除したか

### レビュー・承認フェーズ

- [ ] ステークホルダー全員のレビューを受けたか
- [ ] フィードバックを反映したか
- [ ] 正式な承認を得たか
- [ ] 変更管理プロセスを定義したか

## 用語集

| 用語 | 説明 |
|------|------|
| ステークホルダー | プロジェクトに影響を与える、または影響を受ける関係者 |
| 要求 | ユーザーやビジネスが達成したいこと |
| 要件 | システムが満たすべき条件として定義されたもの |
| 機能要件 | システムが提供すべき機能 |
| 非機能要件 | 性能・セキュリティ・可用性など機能以外の品質特性 |
| RACI | 役割分担フレームワーク。Responsible（実行責任者）、Accountable（説明責任者）、Consulted（相談先）、Informed（報告先） |
| DACI | 意思決定特化フレームワーク。Driver（推進者）、Approver（承認者）、Contributors（貢献者）、Informed（報告先） |
| スコープ | プロジェクトの対象範囲 |
| スコープクリープ | 要件が徐々に増えていく現象 |
| MoSCoW法 | 優先順位付け手法（Must/Should/Could/Won't） |
| MVP | Minimum Viable Product（実用最小限の製品）。最小限の機能で価値を届け、フィードバックを得る手法 |
| PoC | Proof of Concept（概念実証）。本格開発の前に技術的な実現可能性や効果を検証すること |
| V字モデル | 開発フェーズとテストフェーズを対応させたモデル |
| トレーサビリティ | 要件から設計・テストへの追跡可能性 |
| ユースケース | ユーザーとシステムの対話シナリオ |
| ペルソナ | 典型的なユーザー像を具体化したもの |
| RDRA | Relationship Driven Requirement Analysis。要件を4つのレイヤーで整理し、関係性を可視化するモデルベースの要件定義手法。日本発 |

## 実践演習テーマ

### 演習1: レストランの予約システム

**ステークホルダー分析**:
- お店側: オーナー、店長、スタッフ
- お客さん側: 予約する人、予約せずに来店する人
- その他: 近隣店舗？配達業者？

**ヒアリング項目**:
- 現状の予約方法は？（電話？紙？）
- 予約でよくあるトラブルは？
- 繁忙期・閑散期はいつ？
- キャンセルはどう扱っている？

**業務フローの分岐**:
- 予約が満席の場合
- キャンセルの場合
- 無断キャンセルの場合

### 演習2: 社内経費精算システム

**承認フローの条件**:
- 10万円未満: 直属上司の承認のみ
- 10万円以上: 直属上司 → 部長の承認
- 50万円以上: 直属上司 → 部長 → 経理部長の承認
- 経理チェックで不備があれば、申請者に差戻し

**例外ケース**:
- 承認者が休暇中の場合
- 申請者が退職した場合
- 予算超過の場合
- 領収書を紛失した場合

**非機能要件の検討項目**:
- 同時接続数: 月末の繁忙期に何人が使う？
- 可用性: 24時間稼働が必要？営業時間内だけでいい？
- セキュリティ: 経費データは誰が見られる？
- バックアップ: 何年分保存する必要がある？

**段階的リリース計画**:
- 第1フェーズ: 基本的な申請・承認機能
- 第2フェーズ: レポート機能、分析機能
- 第3フェーズ: 会計システム連携、自動仕訳
