# 要件定義 Skill

プロジェクト固有の要件定義方針。「要件定義の教科書」（tan_go238）の知見を体系化。

## このスキルが使われる場面

- 新規プロジェクトの要件定義を行うとき
- ステークホルダー分析・ヒアリングの設計
- 業務フロー（As-Is / To-Be）の作成
- 機能要件・非機能要件の整理
- スコープ管理・優先順位づけ
- 要件定義書のレビュー・品質チェック

## 要件定義の全体プロセス

1. **ステークホルダー分析** — 誰の要求を聞くべきかを特定する
2. **ヒアリング** — 要求を引き出す（表面的な要望の裏にある本当のニーズを掘る）
3. **業務フロー作成** — As-Is（現状）→ To-Be（あるべき姿）を描く
4. **要件の整理** — 機能要件 + 非機能要件を定義し、曖昧な表現を排除する
5. **優先順位づけ** — MoSCoW法でMust/Should/Could/Won'tに分類する
6. **フェーズ分割** — MVP/PoCを定義し、段階的にリリースする計画を立てる
7. **合意形成** — モック・レビュー・サインオフで関係者の合意を取る

## 核心原則

### 要求 ≠ 要件

- **要求**: お客さんの希望・願望（曖昧、主観的）
- **要件**: システムが満たすべき条件（具体的、検証可能）
- 「使いやすくしたい」→「3クリック以内で完了」に変換する

### 手戻りコストの法則

要件定義で1のコストの問題が、設計で5倍、実装で10倍、テストで20倍、リリース後は100倍以上になる。

### 手段と目的を見極める

お客さんの「要望」は手段であることが多い。「それができると何が嬉しいですか？」と聞いて目的を特定する。

### 「あとで決める」は危険

未決事項は「誰が・いつまでに・何を決めるか」を記録し、期限を切る。

## フレームワーク

### RACI / DACI

- **RACI**: タスクの役割分担（Responsible / Accountable / Consulted / Informed）
- **DACI**: 意思決定の役割分担（Driver / Approver / Contributors / Informed）
- 詳細 → `references/fundamentals.md`

### MoSCoW 法

優先順位づけ。Must（〜60%）/ Should（〜20%）/ Could（〜20%）/ Won't。
「全部必須」は「優先順位がついていない」と同じ。
- 詳細 → `references/trouble-prevention.md`

### RDRA

モデルベース要件定義。4レイヤー（システム価値 → 外部環境 → システム境界 → システム）で要件を整理し、関係性を可視化する。
- 詳細 → `references/rdra-framework.md`
