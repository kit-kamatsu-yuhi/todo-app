# プロダクト企画

$ARGUMENTS のプロダクトアイデアを分析し、以下の手順で計画を策定する。

> **`/plan-issue` との使い分け**:
>
> | | `/plan-product` | `/plan-issue` |
> |---|---|---|
> | **起点** | アイデア・プロダクト企画 | 既存の開発 Issue |
> | **スコープ** | プロダクト全体の計画 | 1 つの Issue の実装計画 |
> | **主な作業** | ニーズ検証 → 要件定義 → Issue 起票 → リリース戦略 | UML → API → DB → テスト → タスク分解 |
> | **出力** | プロダクト計画 + GitHub Issue 群 | `plan.md`（設計・タスク分解） |
>
> `/plan-product` が起票した各 Issue の詳細設計は `/plan-issue` に委譲する。

## 参照スキル

- `indie-app-idea` — バーニングニーズ発見・需要検証
- `grok-search` agent — 競合調査・市場トレンド・ユーザーの声収集
- `uml` — ドメインモデリング（Mermaid）
- `db-design` — DB 設計方針
- `api-design` — API 設計方針
- `security` — セキュリティ方針
- `testing` — テスト戦略
- `ci-cd` — CI/CD パイプライン方針
- `issue` — GitHub Issue 起票
- `estimate` — 見積もり
- `start-issue` — Issue 実装プロセス（計画→実装→PR作成）
- `codex-team` — codex sub-agent チームの一括起動
- `linear-walkthrough` — 実装完了後の構造解説ドキュメント生成
- `estimation-methods` — 見積もり手法（ストーリーポイント・三点見積もり・類推見積もり）・バッファ管理・MoSCoW法

## 手順

### 1. ニーズ分析

`indie-app-idea` skill のナレッジを活用し、プロダクトの根本的な価値を検証する。

- 誰のどんな課題を解決するのか（ペインポイントの特定）
- その課題はバーニングニーズか（なくてはならないもの vs あったら便利なもの）
- 既存の代替手段は何か、なぜそれでは不十分か
- ターゲットユーザーの規模と到達可能性

### 2. バーニングニーズ検証

作る前に売れるかを確認する。

- LP やモックアップで事前に顧客の反応を検証する
- 実際に営業・ヒアリングして購入意思を確認する（「欲しい」ではなく「買う」かどうか）
- 競合調査: `grok-search` agent で X/Web から競合・トレンド・ユーザーの声を収集する

### 3. Go/No-Go 判定

以下の基準で実装に進むか判断する:

| 判定 | 基準（いずれか1つ以上を満たす） |
|------|-------------------------------|
| **Go** | LP の CVR が 3% 以上 / ヒアリングで過半数が「金を払う」と回答 / 競合の致命的弱点を1つ以上特定 |
| **No-Go** | 上記をいずれも満たさない → 実装に進まず追加検証を提案する |

- 基準値はプロジェクトに応じて調整する。重要なのは基準を事前に定義し、曖昧な「いけそう」で進まないこと

### 4. 要件定義

Go 判定後、要件を整理する。

- **ユーザーストーリー化**: 検証済みのニーズを「As a / I want / So that」形式に整理する
- **スコープ定義（MoSCoW法）**: `estimation-methods` skill の MoSCoW法を適用し、全機能を以下の4分類に整理する
  - **Must have** — リリース必須。欠くと配布不可（例: ログイン機能、決済処理）
  - **Should have** — 重要だがリリース可能（例: 通知メール、検索フィルター）
  - **Could have** — あると嬉しいが無くても問題なし（例: ダークモード、CSVエクスポート）
  - **Won't have** — 今回は実施しない。将来候補（例: 多言語対応）
  - ステークホルダーと事前合意し、圧縮要求時に「Should have をフェーズ2に延期」という具体的交渉を可能にする
- **非機能要件の洗い出し**: 具体的な数値目標を定義する

| 非機能要件 | 目標値の例 |
|-----------|-----------|
| レスポンスタイム | p95 < 200ms |
| 可用性 | 99.9% |
| 同時接続数 | 1,000 ユーザー |
| データ保持期間 | 3年 |

- **設計方針の決定**:
  - DB: `db-design` skill を参照し、主要エンティティとリレーションを ER 図で整理する
  - API: `api-design` skill を参照し、ベース URL 戦略と主要エンドポイントを洗い出す
  - セキュリティ: `security` skill を参照し、認証・認可・バリデーションの方針を決定する
  - テスト: `testing` skill を参照し、テスト戦略の方針を決定する
  - CI/CD: `ci-cd` skill を参照し、パイプライン構築の方針を決定する
- **ドメインモデリング**: `uml` skill を使って概念モデルを Mermaid で描く
- 要件定義の結果は `raw/issues/` に記録する

### 5. Issue 起票

要件を開発 Issue に分解し、`/issue` スキルで GitHub Issue として起票する。

- 各 Issue の粒度: 1 PR で完結する単位（500行以内を目安）
- Issue 間の依存関係をリンクで表現する
- 非機能要件を対応する Issue に反映する:
  - パフォーマンス目標 → 負荷テスト・パフォーマンス計測 Issue
  - 可用性目標 → 監視・アラート設定 Issue
  - セキュリティ要件 → セキュリティテスト・脆弱性スキャン Issue
- 各 Issue の詳細設計は `/plan-issue` スキルに委譲する

### 5.5. 実装開始

Issue 起票後、`/start-issue` で実装プロセスを開始する。

| 起点 | 状況 | フロー |
|------|------|--------|
| `/plan-product` | アイデア段階、新規プロダクト | ニーズ検証 → 要件定義 → Issue起票 → `/start-issue` |
| `/plan-issue` | 既存プロジェクトで Issue から着手 | 設計 → `/start-issue` |
| Issue URL 入力 | ユーザーが Issue URL + 「進めろ」と指示 | `/start-issue` → 計画 → 実装 → PR |

### 6. 見積もり

`estimation-methods` skill のナレッジを活用し、プロジェクトの状況に応じて見積もり手法を選択する。

| 状況 | 手法 | バッファ算出 |
|------|------|------------|
| 初回見積もり（実績データなし） | 三点見積もり（PERT）または T シャツサイズ | (O + 4M + P) / 6 で期待値算出 |
| 継続見積もり（ベロシティあり） | ストーリーポイント + `/estimate` 連携 | 過去の見積もり誤差率を係数として適用 |
| 過去に類似タスク実績あり | 類推見積もり | 実績ベースで調整 |

- 要件定義段階の見積もりは不確実性が高いため、精度の限界を明示する
- 各 Issue の `/plan-issue` 完了後に見積もりを再評価する
- 不確実性が高いタスクはスパイク（調査タスク）を早期に投入し、本実装見積もり前に不確実性を低下させる
- MoSCoW法のスコープ分類と連動し、Must のみ / Must+Should の段階的見積もりを提示する

### 7. リリース戦略

- **Must を最初にリリースする**
  - バーニングニーズを満たす最小限の機能だけで構成する
  - 「これがないとプロダクトとして成立しない」ものだけに絞る
  - リリースして実際のユーザーの反応・利用状況を観察する
- **リリース後の反応を見てから Nice-to-have を判断する**
  - ユーザーが実際に使っているか、どこで離脱しているかを計測する
  - 要望が多い機能から優先的に追加する
  - 想定と異なる使い方をしている場合は方向転換を検討する
- 依存関係グラフに基づいて実装順序を決定する
- リスクが高い Issue を先に着手する（早期のリスク検証）

### 8. 合意形成

計画の出力後、AskUserQuestion で以下を確認する:

- **スコープ**: MVP に含める機能の範囲は妥当か
- **技術選定**: 採用する技術・ライブラリに懸念はないか
- **優先順位**: 実装順序に異論はないか
- **見積もり**: 見積もり結果とリリース目標に乖離がないか

承認を得るまで計画を確定しない。フィードバックがあれば該当 Step に戻って修正する。
