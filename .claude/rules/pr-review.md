# PR レビュー基準

## PR サイズ制限

- PR は最大 **500行** を目標とする
- 500行を超える場合は分割を検討する
- 自動生成コード（ロックファイル等）は行数カウントから除外する

## レビュー必須項目

- [ ] 要件を満たしているか
- [ ] テストが追加・更新されているか（詳細は `testing` skill 参照）
- [ ] エラーハンドリングが適切か
- [ ] セキュリティ上の問題がないか（詳細は `security` skill 参照）
- [ ] パフォーマンスへの影響がないか
- [ ] 命名規則に従っているか（詳細は `naming` rule 参照）
- [ ] 不要なコード（デバッグログ、コメントアウト等）がないか
- [ ] `wiki/` / `raw/` ドキュメントの更新が含まれているか（最低でも `raw/` に該当 Issue のコンテキストを追加すること）

## レビュー承認

- 最低 **1名** の承認が必要
- 重大な変更（アーキテクチャ変更、セキュリティ関連）は **2名** の承認を推奨

## レビューの進め方

1. まず全体の変更意図を理解する
2. アーキテクチャ・設計レベルの問題を優先的に指摘する
3. セキュリティ観点のチェックは `security` skill の観点に従う
4. テスト観点のチェックは `testing` skill の観点に従う
5. ドキュメント更新（`wiki/` / `raw/`）が含まれているか確認する
6. 細かいスタイルの指摘はリンター・フォーマッターに任せる
7. 指摘は具体的な改善案とともに行う
